二、问题与用户
要解决的真实问题与痛点描述
在高等教育环境下，学生每学期需面对多门课程的“终结性项目（Final Project）”。这些任务通常要求复杂、文档冗长且周期跨度大。
启动焦虑（Analysis Paralysis）：面对老师发布的数页作业 PDF 或截图，学生往往因信息过载而产生心理抗拒，不知从何下手，导致作业一直停留在“待开始”状态。
拆解能力缺失：大部分学生缺乏项目管理经验，无法将一个长期任务科学拆解为每周可执行的“微任务”，导致进度失控，最终在 DDL 前夕进行低质量的“突击”。
环境割裂：现有的商用 AI 虽能生成建议，但缺乏与任务进度绑定的结构化界面，对话记录易丢失，无法提供持续的进度反馈。
使用场景
学生在收到作业要求的瞬间，只需拍照上传给 A-Buddy，系统将直接生成可视化看板和第一步的“起步模板”。这种“即时反馈+结构化引导”能有效降低任务启动的摩擦力，将 AI 的能力从简单的“问答”转化为实用的“执行助手”。
三、需求与范围
核心用户故事
作为一名面临期末压力的学生，我希望通过上传一张“项目计划书模板”的截图，让 AI 自动识别出所有的考核项，并为我创建一个带有时间节点的进度表，以便我清晰地看到每个阶段的目标。
作为一名容易拖延的学生，当我点击“开始写第二章”时，我希望 AI 能结合我之前上传的要求，给我提供 3 个具体的写作思路或初始段落，帮我克服“白纸恐惧症”。
不做什么（Out of Scope）
不提供代写全篇论文或生成完整项目代码的功能，以维护学术诚信。
不涉及多用户社交功能或复杂的课程表同步，仅专注于单一作业的深度拆解。
约束条件
时间：必须在第 14 周前完成从原型到部署的全过程。
工具：受限于免费 API 额度，需在前端或代码逻辑中实现请求控制机制。
隐私：用户上传的作业材料仅用于当前会话处理，不进行云端长期持久化存储。
四、验收标准（AC）
功能性：系统能够准确识别图片中的关键词（如“Deadline”、“要求”、“分值”），并将其转化为结构化的 JSON 任务列表。
功能性：用户可以通过交互界面手动勾选或编辑 AI 生成的任务节点。
质量：生成的任务拆解步骤必须符合逻辑先后顺序（例如：技术选型必须在代码实现之前）。
可用性：系统需提供“进度百分比”可视化展示，方便实时直观查看项目完成进度。
兼容性：应用能够支持移动端 Chrome/Safari 浏览器正常显示，页面布局与按钮适配手指操作。
五、技术方案与选型
技术栈：Python + Streamlit。理由：抛弃繁琐的前端工程化配置，利用 Streamlit 极简的 Python API 快速生成现代化的响应式 Web 看板，最大程度缩短环境搭建周期，实现真正的 Vibe Coding。
AI 集成：直接在 Python 后端调用 Gemini 2.5 Flash API。理由：Gemini 的 Vision 能力在处理长图和复杂表格时表现优异且响应迅速。
替代方案：若 Gemini API 不稳定，将切换为本地运行的 Ollama 或集成 OpenAI 的 GPT-4o-mini，以确保核心功能的连续性。

七、风险清单与应对
模型输出格式不稳定：AI 生成的 JSON 有时会包含多余的 Markdown 标记（如 ```json）。应对：在 Prompt 中强化约束，并编写 Python 正则表达式过滤函数进行容错解析。
作业要求过于模糊：老师给的要求字数太少，AI 无法有效拆解。应对：增加主动追问提示，引导学生手动补充课程背景或自身基础信息。
开发进度滞后：因界面交互调优耗费过多时间。应对：利用 Cursor 的自动代码补全与 Streamlit 自带的现代组件，优先保证核心数据流走通，避免深陷底层样式调整。
